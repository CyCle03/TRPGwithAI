# AI GM 솔로 던전 월드 (MVP) — 작업 브리프

## 목적
혼자서 던전 월드(PbtA)를 플레이할 수 있도록, AI가 GM(게임 마스터)을 맡는 웹 도구를 만든다. 플레이어는 텍스트로 행동을 서술하고, AI GM이 상황을 묘사·전개하며, 주사위 판정과 수치 관리는 코드(규칙 엔진)가 담당한다. HP·인벤토리 등 캐릭터 상태는 화면의 상태창에 항상 표시된다.

이것은 1단계 MVP다. 목표는 "한 캐릭터로 짧은 모험 한 편을 처음부터 끝까지 플레이 가능"한 수준. 멀티플레이·세이브 슬롯 다수·복잡한 룰 자동화는 이후 확장.

## 기술 스택 (기존 프로젝트와 통일)
- Node.js + Express + Socket.io (기존 `computer-upgrade-web`과 동일 계열)
- Claude API (AI GM 두뇌) — Anthropic API 호출. API 키는 서버 환경변수로만 관리하고 절대 클라이언트에 노출하지 않는다. (프런트→자체서버→Anthropic 순 프록시)
- 저장소: 세션 상태 영속화를 위해 경량 저장 사용 (SQLite 또는 기존 Supabase/PostgreSQL). MVP는 단일 세션 저장/불러오기만 지원.
- 배포: 기존 오라클 ARM 인스턴스 (Docker 컨테이너 추가, 기존 서비스 불간섭)

## 핵심 설계 원칙 — 역할 분리 (이 프로젝트의 심장)
규칙 엔진(코드)과 AI GM(LLM)의 책임을 명확히 나눈다. 이유: LLM에게 주사위·수치까지 맡기면 계산을 틀리거나 규칙을 무시하는 일이 잦다. 서사는 LLM이 잘하지만, 판정의 공정성·일관성은 결정론적 코드가 보장해야 한다.

### 코드(규칙 엔진)가 담당
- 주사위 굴림: 2d6 난수 생성 (반드시 서버에서).
- 판정 계산: 2d6 + 능력치 보정 → 10+ / 7–9 / 6- 구간 결정.
- 상태 관리: HP, 능력치, 인벤토리, 상태이상 등 수치의 단일 원천(source of truth). LLM이 말한 걸 상태로 삼지 않는다 — 상태 변경은 코드가 검증 후 반영.
- 판정 결과를 구조화해서 AI GM에게 전달.

### AI GM(Claude API)가 담당
- 상황·장면 묘사, NPC 대사, 세계관 연출.
- 플레이어의 자유서술을 던전 월드 무브로 유연하게 해석 (예: "몰래 다가간다" → Defy Danger(DEX)). 어떤 무브이고 어떤 능력치를 쓰는지 판단만 하고, 실제 주사위는 코드에 요청한다.
- 판정 결과(코드가 굴린 값)를 받아 서사로 풀어낸다 (10+ 성공 서사 / 7–9 대가 있는 성공 / 6- 실패와 악화).
- 상태 변경이 필요하면(예: "5 데미지") 구조화된 지시로 코드에 요청, 코드가 반영.

## AI GM ↔ 규칙 엔진 통신 설계
LLM이 자유 텍스트만 뱉으면 코드가 처리할 수 없으므로, AI GM의 응답을 서사 텍스트 + 구조화된 액션(JSON)으로 분리해서 받는다.

권장 방식: Claude API 호출 시 시스템 프롬프트로 "응답을 두 부분으로 나눠라 — 플레이어에게 보여줄 서사, 그리고 시스템이 실행할 액션(JSON)"이라고 지시. 예시 액션 스키마:

```json
{
  "narration": "너는 숨을 죽이고 고블린 뒤로 다가간다...",
  "action": {
    "type": "roll",              // roll | update_state | none
    "move": "Defy Danger",
    "stat": "DEX",
    "reason": "기습을 시도"
  }
}
```

- `type: "roll"` → 코드가 2d6+해당 stat 보정 굴림 → 결과를 다시 AI GM에 전달하여 그 결과에 맞는 서사를 생성 (2-패스: 판정요청 → 굴림 → 결과서사).
- `type: "update_state"` → 코드가 HP/인벤토리 등 변경 (검증 후 상태창 갱신).
- `type: "none"` → 순수 대화/묘사, 판정 불필요.

이 "AI가 판정을 요청하면 코드가 굴리고 결과를 다시 AI에 먹여 서사화"하는 루프가 MVP의 가장 중요한 기술 포인트다. 여기를 견고하게 만든다.

## 상태(캐릭터) 모델 (상태창에 표시)
MVP 캐릭터 상태 최소 스키마:
- 이름, 클래스(던전 월드 기본 클래스 중 택1 — MVP는 1~2개만 지원해도 됨)
- 능력치 6종과 보정: STR/DEX/CON/INT/WIS/CHA (각 보정 -1~+3)
- HP (현재/최대), 방어구(armor)
- 인벤토리 (아이템 목록, 간단한 배열)
- (확장 여지) 상태이상, 경험치, 유대(bonds), 무브 목록

상태창 UI: 화면 한쪽에 고정 패널로 HP·능력치·인벤토리 상시 표시. 값이 바뀌면(피해/회복/아이템 획득) 실시간 갱신 + 변경 강조.

## 화면 구성 (MVP)
- 좌/중앙: 서사 로그 — AI GM 묘사와 플레이어 입력이 채팅처럼 시간순으로 쌓임. 주사위 결과는 로그 안에 구분되게 표시 (예: 🎲 2d6+1 = [4,5]+1 = 10 → strong hit).
- 우측: 상태창 패널 — HP 바, 능력치, 인벤토리. 항상 보임.
- 하단: 입력창 — 플레이어가 행동을 자유 서술. 전송 시 AI GM 처리.
- 처음 진입 시: 캐릭터 생성(간단히 — 클래스 선택 + 능력치 배분 또는 프리셋) 후 모험 시작.

## 기능 범위
### 필수 (MVP)
1. 캐릭터 생성 (최소: 이름 + 클래스 프리셋 1개 + 기본 능력치).
2. AI GM과 자유 텍스트 대화로 장면 진행.
3. AI GM이 무브를 판단 → 코드가 2d6 판정 → 결과를 AI GM이 서사화 (핵심 루프).
4. HP·인벤토리 상태창 실시간 표시·갱신.
5. 세션 저장/이어하기 (최소 1슬롯). 서버 재시작 후에도 복구.

### 이번엔 제외 (과설계 방지)
- 멀티플레이(솔로 전용). 로그인/계정.
- 전체 클래스·전체 무브 구현 (1~2 클래스로 시작).
- 이미지·음성·지도.
- 정교한 전투 자동화(적 AI 턴 관리 등) — MVP는 GM 서사 중심으로 가볍게.

## Claude API 사용 주의
- API 키는 서버 환경변수(.env)로만. 클라이언트 코드/네트워크 응답에 절대 포함 금지.
- 프런트는 자체 서버에만 요청하고, 서버가 Anthropic API를 대신 호출(프록시).
- 컨텍스트 관리: 세션이 길어지면 전체 대화를 매번 보내면 토큰이 커진다. 최근 N턴 + **코드가 관리하는 구조화 상태 요약(현재 HP/장소/주요 NPC/목표)**을 시스템 프롬프트에 넣어 GM이 맥락을 잃지 않게 한다. (LLM 기억의 한계를 코드가 보완)
- 모델·토큰 한도 등은 현재 Anthropic 공식 문서를 확인해 설정.

## 배포 (기존 오라클 서버)
- 기존 웹 게임·AdGuard·VPN 도는 서버에 Docker 컨테이너로 추가.
- 내부 포트 하나 할당(기존과 비충돌) → 기존 nginx 리버스 프록시에 서브도메인/경로 추가. Socket.io 사용 시 nginx WebSocket 업그레이드 헤더 필수.
- SSL은 기존 certbot 인증서 재사용/서브도메인 추가.
- nginx 경유이므로 새 포트를 오라클 Security List에 별도로 열 필요 없음.

## 검증 시나리오
1. 캐릭터 생성 → 모험 시작, AI GM이 첫 장면을 묘사하는지.
2. "고블린을 몰래 기습한다" 입력 → AI가 무브/능력치를 판단하고 코드가 주사위를 굴린 뒤 그 결과(10+/7-9/6-)에 맞는 서사가 나오는지. (핵심 루프 동작 확인)
3. 주사위 숫자·보정·구간이 규칙대로 정확한지 (코드가 굴렸으므로 항상 일관).
4. 피해를 입는 상황 → HP 상태창이 실제로 감소하는지 (AI 말과 상태창 수치 일치).
5. 아이템 획득 → 인벤토리에 반영되는지.
6. 저장 후 서버 재시작 → 이어하기로 상태·진행이 복구되는지.
7. 긴 세션에서 AI가 앞선 맥락(획득 아이템/HP/NPC)을 유지하는지.

## 확장 로드맵 (이번엔 구현 안 함)
- 2단계: 던전 월드 전체 기본 클래스 지원, 무브 목록 참조(원문 복제 금지, 직접 요약).
- 3단계: 전투 시 적 스탯/턴 관리 반자동화, 상태이상·유대·경험치.
- 4단계: 세이브 슬롯 다수, 세션 요약/회고 생성, 분위기(톤) 프리셋.
- 5단계: 멀티플레이(친구와 함께 + AI GM).

## 저작권 주의
던전 월드의 규칙 메커니즘(2d6 판정 등)은 자유롭게 구현 가능하나, 룰북의 무브 설명·텍스트를 그대로 복제하지 않는다. 무브/클래스 설명은 직접 요약·재작성하고, AI GM 프롬프트에도 원문을 통째로 넣기보다 요지를 넣는다. 던전 월드는 상당 부분이 Creative Commons로 공개돼 있으니 배포 전 라이선스 확인.
